🛡️ Riscos & Mitigações

Análise completa de riscos identificados no sistema Nommand Vision e as mitigações implementadas. MITIGADO

✅ Resumo das Mitigações Implementadas

  • Rate Limit por Workspace — Proteção contra DoS por workspace
  • Blacklist de Tokens JWT — Logout imediato, revogação instantânea
  • SQL Injection Prevention — Validação strict de parâmetros de query
  • Cross-Workspace Isolation — Verificação de workspace em todas as queries
  • Prompt Sandbox — Filtro contra injection em system prompts
  • Token Rate Limit — Limite de tokens Anthropic por workspace/min

🔴 Riscos Críticos (Alto)

Rate Limit Insuficiente ALTO

Auth endpoints tinham rate limit apenas por IP, sem isolamento por workspace. Possível ataque DoS usando um único workspace para esgotar limites compartilhados.

✅ Mitigação Implementada
  • Rate limit por IP (fallback)
  • Rate limit por workspace (300 req/min)
  • Rate limit por usuário (120 req/min)
  • Headers X-Workspace-RateLimit-Remaining e X-User-RateLimit-Remaining
workspaceRateLimit, apiRateLimit, agentRateLimit
JWT sem Revogação Imediata ALTO

Access tokens não tinham blacklist. Logout não invalidava tokens imediatamente, permitindo uso por até 15 minutos após revogação.

✅ Mitigação Implementada
  • Tabela token_blacklist criada
  • Hash SHA-256 de tokens armazenados
  • Verificação em verifyAccessToken()
  • Invalidacao em logout e troca de senha
tokenBlacklistService.addToBlacklist(token, type, expiresAt)

🟡 Riscos Moderados (Médio)

SQL Injection via JSON MÉDIO

Queries dinâmicas com JSONB e filtros do usuário poderiam ser vulneráveis se não validadas. Drizzle protege, mas queries dinâmicas ainda requerem cuidado.

✅ Mitigação Implementada
  • Serviço QueryValidatorService com Zod validation
  • Whitelist de colunas permitidas para ordenação
  • Whitelist de filtros permitidos
  • Validação strict de UUIDs, enums, datas
validateQueryParams(params): { sort, page, limit, filters }
Cross-Workspace Data Leak MÉDIO

Isolamento poderia falhar em edge cases. Middleware verifica workspaceId, mas queries manuais podem esquecer o filtro.

✅ Mitigação Implementada
  • Todas as queries já incluem workspaceId no where
  • Middleware adiciona c.set('workspaceId')
  • Services sempre recebem workspaceId como primeiro parâmetro
  • Verificação explícita em endpoints com params (ex: get agent por id)
if (agent.workspaceId !== user.workspaceId) return 404
Agent Prompt Injection MÉDIO

Usuários controlam systemPrompt em agentes customizados. Podem injetar instruções maliciosas para contornar comportamentos.

✅ Mitigação Implementada
  • PromptSandboxService valida prompts antes de enviar à Claude
  • Detecção de padrões perigosos: "ignore previous instructions", "system prompt"
  • Sanitização de newlines, remoção de blocos de código
  • Whitelist de commands proibidos
validateAndSanitize(prompt, agentType) → { isValid, sanitized }
Custos de IA Descontrolados MÉDIO

Sem limites, agentes podem gerar custos excessivos na Anthropic API se configurados com maxTokens alto ou loops ineficientes.

✅ Mitigação Implementada
  • TokenRateLimitService com limite de 50.000 tokens/min/workspace
  • Limite por usuário: 10.000 tokens/min
  • Limite por agente: 20.000 tokens/min
  • Cache em memória + consulta a DB para precisão
canUseTokens(workspaceId, estimatedTokens) → { allowed, remaining }

🟢 Riscos Baixos

Logs com Dados Sensíveis BAIXO

Logs podem conter tokens JWT, PII ou dados sensíveis se não sanitizados.

⚠️ Mitigação Pendente
  • Implementar logger com redaction automático
  • Sanitizar senhas, tokens, emails dos logs
  • Auditoria de logs existentes
TODO: logger.sanitize(message)
Deploys Falhos Silenciosos BAIXO

Deploys podem falhar sem notificação adequada, deixando usuário sem saber do erro.

⚠️ Melhoria Sugerida
  • Notificações obrigatórias para falhas
  • Status dashboard em tempo real
  • Webhook para alertas externos
deployService.sendNotification(status, error)
Webhooks sem Retry BAIXO

Webhooks externos podem falhar sem retry automático, causando perda de eventos.

⚠️ Melhoria Sugerida
  • Dead letter queue para falhas
  • Retry exponencial (5 tentativas)
  • Backoff exponencial (1s, 2s, 4s...)
  • Dead letter alert após exaurir retries
webhookQueue.retry(job, maxRetries=5)

🔧 Como Testar as Mitigações

  1. Rate Limit: Faça 130 requisições em 1 minuto no mesmo workspace — deve receber 429 após 120.
  2. JWT Blacklist: Faça login, logout, e tente usar o token anterior — deve receber 401.
  3. Prompt Sandbox: Envie prompt contendo "ignore previous instructions" — deve receber erro 400.
  4. Token Limit: Simule uso de 51.000 tokens em 1 min — deve receber 429.
  5. Cross-Workspace: Tente acessar recurso de outro workspace — deve receber 404.
