Workspaces & Multitenancy

Sistema completo de isolamento multi-tenant com workspaces hierárquicos, planos diferenciados e delegação de usuários. Garante segurança total entre tenants através de foreign keys cascading e middleware.

Propósito

🏢

Isolamento Multi-Tenant

Cada workspace representa um tenant (organização/equipe/projeto). Todos os dados são isolados por workspaceId. Usuários de um workspace NÃO podem acessar dados de outro workspace.

📊

Planos & Limites

Sistema de planos: free, pro, enterprise. Define limites de uso, features habilitadas e cotas por organização.

🔗

Estrutura Hierárquica

Workspace possui owner (criador) e múltiplos usuários membros. Relação 1-N: 1 workspace pode ter N usuários. Deleção em cascata: deletar workspace remove todos os usuários vinculados.

Conceitos Chave

  • Tenant: Um workspace/cliente organizacional
  • Owner: Usuário que criou o workspace
  • Membro: Usuário vinculado ao workspace
  • Cascading Delete: Ao deletar workspace, todos os usuários são removidos automaticamente pelo banco
  • workspaceId: UUID único que identifica o tenant em todas as queries

4

→
3

Schemas do Banco de Dados

🗄️

Tabela: workspaces

Tabela principal de workspace. Cada registo representa uma organização/tenant.ownerId referencia o usuário criador, mas sem FK explícita.

👥

Tabela: users

Usuários vinculados a workspaces. workspaceId é NOT NULL com FK (ON DELETE CASCADE). Dele workspace = deleta todos os users automaticamente.

⏰

Tabela: sessions

Refresh tokens armazenados com userId FK (cascade). Dele user = limpa todas as sessões.

Identificação do Workspace

Workspace NÃO possui rotas API próprias. É identificado automaticamente via middleware stack:

Execution Order: auth + workspace

1. authMiddleware (executa primeiro)
└─ Lê Authorization header → JWT
└─ Busca user no DB por userId
└─ Extrai user.workspaceId
└─ c.set('user', { id, email, workspaceId })
2. workspaceMiddleware (executa depois)
└─ Lê user.workspaceId do contexto
└─ Valida se existe
└─ c.set('workspaceId', user.workspaceId)
└─ Rejeita se não houver workspace
3. Handler
└─ Recupera: const workspaceId = c.get('workspaceId')
└─ Usa em queries: eq(item.workspaceId, workspaceId)

Segurança & Isolamento

Workspace isolation é a base da arquitetura multi-tenant. Violação causa vazamento catastrófico de dados.

ALTO

Query sem workspaceId

Se um handler esquece de filtrar por workspaceId, todos os tenants retornam dados de todos. Vazamento total.

Mitigação: 1. Workspace middleware garante workspaceId no contexto 2. Code review obrigatório: buscar workspaceId em TODAS as queries 3. Helper: db.item.findMany({ where: { workspaceId } }) 4. Considerar query interceptor automático
MÉDIO

ownerId como owner

Workspace.ownerId não tem FK explícita. Owner pode ser deletado sem efeito no workspace, deixando orfão.ownerId seria null? Não, atualmente não cascade.

Mitigação: - Dono não deve ser deletado (desativar conta em vez) - Ou adicionar FK: .references(() => users.id) - Negócio: owner é o usuário inicial criador
BAIXO

Race condition em workspaces.id

uuid defaultRandom() é umico global, não colisão.

Mitigação: já seguro
ALTO

slug UNIQUE mas sem validação

Slug é UNIQUE, mas pode haver conflito em integrações. Insert não tratado para slug duplicado.

Mitigação: - Catch DB unique constraint error - Retornar 409 CONFLICT - Sugerir slug alternativo

Medidas de Isolamento Implementadas

  • Foreign keys workspaceId em todas as tabelas de dados
  • CASCADE DELETE em workspaceId (deleta workspace = deleta tudo)
  • Middleware workspace valida workspaceId em cada request
  • workspaceId incluído automaticamente pelo authMiddleware
  • Todas as queries DEVEM incluir eq(workspaceId, workspaceId)
  • Cada usuário pertence a exatamente 1 workspace
  • JWT payload contém userId apenas (workspaceId buscado no DB)
  • schema.users.chave estrangeira CASCADE via Drizzle ORM

Integrações

Workspaces não tem endpoints próprios. É integrado em outros módulos através do middleware stack.

🔗

authMiddleware

Primeiro na chain. Valida JWT, busca user no DB (inclui workspaceId), define c.set('user', { id, email, workspaceId }). Sem workspaceId, autenticação falha na próxima stage.

🏢

workspaceMiddleware

Segundo na chain. Lê workspaceId de user, valida se presente, define c.set('workspaceId', user.workspaceId). Rejeita se user sem workspace (401).

📦

Drizzle ORM + Workspace Filters

Schema users tem workspaceId: uuid().references(...).onDelete('cascade'). Todas as queries devem fazer eq(relations.workspaceId, workspaceId).

🗄️

PostgreSQL Cascade

ON DELETE CASCADE nas FKs: users.workspaceId → workspaces.id. Deleção de workspace limpa todos os users automaticamente no nível do banco.

👥

Users Table como Bridge

A tabela users é a ponte workpace ↔ usuários. Possui workspaceId NOT NULL com FK cascade. Define o relacionamento 1:N.

📝

Context Type System

AppEnv.Variables inclui workspaceId garantindo type safety. c.get('workspaceId') sempre retorna string (não null após workspaceMiddleware).

Middleware Chain Aplicação

Todo módulo que requer autenticação e workspace isolation deve aplicar ambos os middlewares na ordem:

// Exemplo em um router
import { authMiddleware } from '../middleware/auth.js'
import { workspaceMiddleware } from '../middleware/workspace.js'

router.get(
  '/items',
  // Ordem importa: workspaceMiddleware depende de authMiddleware
  authMiddleware,
  workspaceMiddleware,
  async (c) => {
    const workspaceId = c.get('workspaceId') // garantido
    // ... queries com workspaceId filter
  }
)

Como Usar

Workspaces são criados implicitamente no registro.开发者 não chamam rotas workspaces diretamente. Usam workspaceId do contexto.

Exemplo: Criar Item em um Workspace

// Handler com auth + workspace middleware export const createItem = async (c: Context) => { const { workspaceId } = c.req.valid('json') // validado por zod const userId = c.get('user').id // workspaceId do contexto NÃO do payload! // workspaceId já veio do middleware (auth → DB lookup) const ctxWorkspaceId = c.get('workspaceId') // Query com workspace filter garantido const [item] = await db.insert(items).values({ name: req.name, workspaceId: ctxWorkspaceId, // ISOLATION: sempre filter createdBy: userId, }).returning() }

Fluxo Cliente: Obter workspaceId

1. Login (POST /api/auth/login) → Recebe accessToken + refreshToken → accessToken contém userId → NÃO contém workspaceId (é lookup do DB) 2. Requisição para módulo protegido GET /api/items Header: Authorization: Bearer {accessToken} 3. Backend processa: [auth] extrai userId do JWT [auth] busca user no DB, obtém workspaceId [workspace] valida e seta c.workspaceId [handler] usa workspaceId na query 4. Response: apenas items do workspace do user

Regra de Ouro para Developers

NUNCA confie em workspaceId do request body.

SEMPRE use c.get('workspaceId') do contexto (já validado pelo auth+workspace middleware).

SEMPRE filtrar queries por eq(..., workspaceId).

Exceção: quando workspaceId precisa ser alterado (ex: admin role) — requer validação adicional de permissões.

Schema SQL Consolidado

-- Enum Plan
CREATE TYPE plan AS ENUM ('free', 'pro', 'enterprise');

-- Enum Role
CREATE TYPE role AS ENUM ('Frontend', 'Backend', 'Design', 'Data', 'Admin');

-- Enum Theme
CREATE TYPE theme AS ENUM ('dark', 'light');

-- Table: workspaces
CREATE TABLE workspaces (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name VARCHAR(100) NOT NULL,
  slug VARCHAR(100) NOT NULL UNIQUE,
  ownerId UUID NOT NULL,
  plan plan NOT NULL DEFAULT 'free',
  createdAt TIMESTAMP NOT NULL DEFAULT NOW(),
  updatedAt TIMESTAMP NOT NULL DEFAULT NOW()
);

-- Table: users com FK cascade
CREATE TABLE users (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  workspaceId UUID NOT NULL REFERENCES workspaces(id) ON DELETE CASCADE,
  email VARCHAR(255) NOT NULL UNIQUE,
  passwordHash VARCHAR(255),
  name VARCHAR(100) NOT NULL,
  avatarUrl VARCHAR(500),
  role role,
  stack JSONB,
  cpuLevel INTEGER,
  onboarded BOOLEAN NOT NULL DEFAULT FALSE,
  theme theme NOT NULL DEFAULT 'dark',
  tourDone BOOLEAN NOT NULL DEFAULT FALSE,
  plan user_plan NOT NULL DEFAULT 'free',
  oauthProvider VARCHAR(20),
  oauthId VARCHAR(255),
  emailVerified BOOLEAN NOT NULL DEFAULT FALSE,
  createdAt TIMESTAMP NOT NULL DEFAULT NOW(),
  updatedAt TIMESTAMP NOT NULL DEFAULT NOW()
);

-- Index para queries por workspace (performance)
CREATE INDEX idx_users_workspaceId ON users(workspaceId);

-- Table: sessions (refresh tokens)
CREATE TABLE sessions (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  userId UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  refreshTokenHash VARCHAR(255) NOT NULL,
  device VARCHAR(100),
  browser VARCHAR(50),
  ip VARCHAR(45),
  expiresAt TIMESTAMP NOT NULL,
  createdAt TIMESTAMP NOT NULL DEFAULT NOW()
);

Migration Reference (Drizzle)

O schema é definido em /backend/src/db/schema/workspaces.ts e /backend/src/db/schema/users.ts:

// workspaces.ts
export const workspaces = pgTable('workspaces', {
  id: uuid('id').primaryKey().defaultRandom(),
  name: varchar('name', { length: 100 }).notNull(),
  slug: varchar('slug', { length: 100 }).notNull().unique(),
  ownerId: uuid('owner_id').notNull(),
  plan: planEnum('plan').notNull().default('free'),
  createdAt: timestamp('created_at').notNull().defaultNow(),
  updatedAt: timestamp('updated_at').notNull().defaultNow(),
})

// users.ts (trecho workspace)
workspaceId: uuid('workspace_id').notNull().references(() => workspaces.id, {
  onDelete: 'cascade'
}),

Para aplicar migrations: drizzle-kit generate e drizzle-kit migrate.
