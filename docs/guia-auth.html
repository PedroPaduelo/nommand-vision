Módulo de Autenticação & Usuários

Sistema completo de autenticação JWT com refresh tokens, gerenciamento de usuários e workspaces. Implementado com Hono, Drizzle ORM e PostgreSQL.

O que é

🔐

Propósito

Gerenciar autenticação, autorização e ciclo de vida de usuários na plataforma. Fornece endpoints para registro, login, logout, refresh de tokens e gerenciamento de perfil.

🏢

Workspaces

Cada usuário pertence a um workspace. O workspace criado durante o registro representa a organização/equipe do usuário. O ownerId do workspace aponta para o usuário que criou.

👥

Atores

Usuário: Pessoa que se registra e usa a plataforma.
Sistema: Backend que valida credenciais, emite tokens, gerencia sessões.
Middleware Auth: Intercepta requisições e valida tokens JWT.

Tratamento de Erros

Todos os erros retornam o formato {"error": {"code": "...", "message": "..."}} com status HTTP adequado.

Código Status HTTP Quando Ocorre Exemplo
NOT_FOUND 404 Usuário não existe ao acessar /me {"error":{"code":"NOT_FOUND","message":"User not found"}}
CONFLICT 409 Email já registrado em /register {"error":{"code":"CONFLICT","message":"User already exists"}}
UNAUTHORIZED 401 Credenciais inválidas, token expirado, header ausente, usuário não encontrado {"error":{"code":"UNAUTHORIZED","message":"Invalid email or password"}}
INTERNAL_ERROR 500 Erro inesperado no register, login, update, etc. {"error":{"code":"INTERNAL_ERROR","message":"Registration failed"}}

Segurança & Riscos

ALTO

Vazamento de Token JWT

Tokens JWT são auto-contidos. Se vazarem, podem ser usados até expirar.

Mitigação: - Access token: 15 minutos - Refresh token: 7 dias - Refresh tokens armazenados em DB (sessions) - Refresh token pode ser revogado individualmente - Logout remove sessão do DB - HTTPS obrigatório em produção
MÉDIO

Senhas em Texto Plano

Armazenar senhas sem hash é risco crítico de segurança.

Mitigação: - bcrypt com salt rounds configurável - Hash é calculado assincronamente - passwordHash é nullable (suporta OAuth) - Nunca retorna hash na API
MÉDIO

Race Condition em Refresh

Múltiplas requisições de refresh simultâneas podem criar sessões duplicadas.

Mitigação: - Refresh token hash é único na combinação userId+hash - Delete antes do insert no refresh - Considere DB constraint UNIQUE(userId, refreshTokenHash)
ALTO

Brute Force no Login

Atacantes podem tentar combinações de senha.

Mitigação: - Rate limit: authRateLimit middleware - Implementado em middleware/rate-limit.ts - Limite por IP + usuário - Considerar adicionar exponential backoff
MÉDIO

Session Fixation

Reutilizar sessão antiga após login pode manter sessões roubadas.

Mitigação: - Sessões são criadas apenas após login/refresh - Logout deleta todas as sessões do usuário - Refresh token é rodado a cada uso (rotaciona)
BAIXO

SQL Injection

Drizzle ORM usa query builder com parâmetros preparados.

Mitigação: - Uso de Drizzle ORM (prepared statements) - Validação com Zod (zValidator) - Nenhum SQL concatenação direta

Medidas de Segurança Implementadas

  • Rate limiting (authRateLimit)
  • Bcrypt para hash de senhas
  • JWT com assinatura HS256 (library: jose)
  • Refresh tokens persistidos em DB (sessions)
  • Validação Zod para todos os inputs
  • CORS configurado (no app principal)
  • Tokens com expiração curta (access) e longa (refresh)
  • Rotação de refresh token em cada uso
  • Cascade delete de sessões ao deletar usuário

Como Usar (cURL Examples)

1. Registrar Usuário

curl -X POST http://localhost:3000/api/auth/register \ -H "Content-Type: application/json" \ -d '{"email":"dev@nommand.com","password":"minhaSenhaSegura123","name":"João Dev"}'

2. Login

curl -X POST http://localhost:3000/api/auth/login \ -H "Content-Type: application/json" \ -d '{"email":"dev@nommand.com","password":"minhaSenhaSegura123"}'

3. Acessar /me

curl http://localhost:3000/api/auth/me \ -H "Authorization: Bearer SEU_ACCESS_TOKEN"

4. Atualizar Perfil

curl -X PATCH http://localhost:3000/api/auth/me \ -H "Authorization: Bearer SEU_ACCESS_TOKEN" \ -H "Content-Type: application/json" \ -d '{"role":"Backend","stack":["TypeScript","Node.js"],"theme":"dark"}'

5. Refresh Token

curl -X POST http://localhost:3000/api/auth/refresh \ -H "Content-Type: application/json" \ -d '{"refreshToken":"SEU_REFRESH_TOKEN"}'

6. Logout

curl -X POST http://localhost:3000/api/auth/logout \ -H "Authorization: Bearer SEU_ACCESS_TOKEN" \ -H "Content-Type: application/json" \ -d '{"refreshToken":"SEU_REFRESH_TOKEN"}'

Integrações

🔒

Middleware Auth

Definido em /backend/src/middleware/auth.ts. Protege rotas adicionando c.set('user', { id, email, workspaceId }) ao contexto Hono.

⏱️

Rate Limit

Middleware authRateLimit aplicado em todos os endpoints auth. Implementado em /backend/src/middleware/rate-limit.ts.

🗄️

Drizzle ORM + DB

Conexão compartilhada via db singleton. Schema em /backend/src/db/schema/. Queries com eq, and.

🔑

JWT (jose)

Biblioteca jose. Funções signAccessToken, signRefreshToken, verifyAccessToken, verifyRefreshToken em /backend/src/lib/jwt.ts.

🔐

Hash (bcrypt)

Funções hashPassword, verifyPassword, hashToken (para refresh tokens) em /backend/src/lib/hash.ts.

📦

Zod Validation

Schemas em /backend/src/schemas/auth.schema.ts. Validação via zValidator do @hono/zod-validator.

Como Outros Módulos Usam

Qualquer módulo que precise do usuário autenticado deve:

  1. Importar authMiddleware
  2. Aplicar nas rotas: router.get('/rota', authMiddleware, handler)
  3. Acessar: const user = c.get('user') para obter { id, email, workspaceId }
  4. Usar workspaceId para queries isoladas por tenant (multi-tenancy)
