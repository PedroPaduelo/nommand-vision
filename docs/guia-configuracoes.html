Módulo de Configurações & Perfil

Sistema completo de gerenciamento de configurações de workspace, perfil de usuário, API keys, integrações externas, aparência, notificações e billing. Implementado com React (frontend) e Hono + Drizzle (backend).

O que é

⚙️

Propósito

Centralizar todas as configurações da plataforma: dados do workspace, preferências do usuário, chaves de API, integrações com serviços externos, aparência da UI, preferências de notificações e gerenciamento de faturamento.

👤

Perfil do Usuário

O usuário pode editar seu nome, bio, avatar, role, stack e preferências de tema. A senha pode ser alterada via endpoint específico. O onboarding e tour podem ser marcados como completos.

🏢

Configurações do Workspace

Configurações globais do workspace: nome, região, domínio personalizado, modo desenvolvedor, auto-save, telemetria. Apenas workspace owners podem alterar algumas configurações sensíveis.

🔑

API Keys

Gerenciamento de chaves de API para acesso programático à plataforma. Criação de novas chaves, cópia segura (mas nunca exibe chave completa após criação), e revogação de chaves antigas.

🔗

Integrações

Conexões com serviços externos: GitHub, GitLab, etc. Cada integração pode ser conectada/desconectada. Inclui webhooks personalizáveis com eventos e segredos.

🎨

Aparência

Personalização visual: tema (dark/light), cor de destaque, sidebar compacta, animações, fonte e tamanho da fonte.

🔔

Notificações

Preferências de quais eventos geram notificações: deploys (sucesso/falha), menções, sugestões de IA, alertas de performance e email digest.

💳

Billing

Gestão de plano, faturamento e uso de recursos: visualização do plano atual, preço, features, uso de storage/bandwidth/membros/tokens, próximo pagamento e método de pagamento.

Fluxo de Dados

1. Atualizar Configuração Individual

1
Frontend Usuário altera toggle (ex: devMode)
→
2
Service updateSetting('devMode', false)
→
3
API PATCH /api/settings/devMode
→
4
Settings Service settingsService.updateSetting(workspaceId, key, value)
→
5
DB UPDATE settings SET value = ?, updatedAt = NOW() WHERE workspaceId = ? AND key = ?
→
6
Resposta { key, value } atualizado

2. Criar API Key

1
Frontend Clica "New API Key", insere nome
→
2
Service createApiKey({ name, permissions, expiresAt })
→
3
API POST /api/settings/api-keys
→
4
Service Gera chave criptográfica (crypto.randomBytes) Hash para storage
→
5
DB INSERT INTO api_keys (id, workspaceId, name, keyHash, permissions, status, createdAt)
→
6
Resposta Retorna chave COMPLETA pela única vez! { id, name, key, status: 'active', createdAt }

3. Atualizar Perfil

1
Frontend Usuário altera nome, bio, stack
→
2
Service updateProfile({ name, bio, stack, cpuLevel })
→
3
API PATCH /api/profile
→
4
Profile Service UPDATE users SET name=?, bio=?, stack=?, cpuLevel=? WHERE id = ?
→
5
Resposta Usuário atualizado + getProfile() para dados formatados

4. Alterar Senha

1
Frontend Form com oldPassword + newPassword
→
2
Service changePassword(oldPassword, newPassword)
→
3
API POST /api/profile/password
→
4
Profile Service Busca usuário → verifica oldPassword com bcrypt
→
5
Hash + Update hashPassword(newPassword) → UPDATE passwordHash
→
6
Resposta { success: true } ou erro se senha inválida

Segurança & Riscos

ALTO

Vazamento de API Keys

Chaves de API são geradas e retornadas apenas uma vez. Se o usuário fechar a janela sem copiar, precisa gerar nova. Se vazar em logs ou exposição acidental, pode dar acesso total à API.

Mitigação: - Chave completa retornada APENAS na criação - Frontend masca chaves após cópia (12 chars + ... + 4) - Nunca armazenar chaves em localStorage - Revogação manual disponível - Considerar expiry date opcional
MÉDIO

Race Condition em Settings

Múltiplos PATCH simultâneos em diferentes chaves podem sobrescrever configurações. Como cada atualização é独立, a ordem pode ser imprevisível.

Mitigação: - Atualizações são por chave individual (atomicidade) - Considerar versão/timestamp para geral settings - Ou enviar objeto completo com PATCH /settings
ALTO

Billing Data Exposure

Informações sensíveis de pagamento (método parcial) são retornadas. Se não houver rate limit, atacante pode enumerar workspaces.

Mitigação: - Auth middleware protege todas rotas - Retornar apenas dados da billing do próprio workspace - Rate limit por usuário - Não expor dados sensíveis de outros usuários
MÉDIO

Webhook URL Injection

URL de webhook é validada com z.string().url() mas pode apontar para servidores internos (SSRF) se allowed.

Mitigação: - Zod valida formato URL - Considerar blacklist de IPs privados - Test endpoint só envia payload mock - Log failures para auditoria
BAIXO

Theme Enum Bypass

Tema aceita apenas dark|light via schema, mas frontend pode usar valores arbitrários se burlar validação.

Mitigação: - Schema valida estritamente - Frontend limita options em select dropdowns
MÉDIO

Password Reset Sem Confirmação

changePassword não exige confirmação da senha atual se o token estiver válido? (Verificar implementação).

Mitigação: - Exige oldPassword sempre - bcrypt verificação async - Log de mudança de senha para auditoria - Forçar logout dos outros dispositivos após mudança

Medidas de Segurança Implementadas

  • Auth middleware em todas rotas
  • Validação Zod em todos inputs
  • Chaves API never exposed após criação
  • Hash bcrypt para senhas
  • Workspace isolation (multi-tenancy)
  • Rate limiting por workspace
  • Logs de ações sensíveis (excluir conta, mudar senha)
  • Webhook URL validation
  • Accent colors apenas predefined set

Como Usar (cURL Examples)

1. Buscar Configurações do Workspace

curl http://localhost:3000/api/settings \ -H "Authorization: Bearer SEU_ACCESS_TOKEN"

2. Atualizar Configuração Específica

curl -X PATCH http://localhost:3000/api/settings/devMode \ -H "Authorization: Bearer SEU_ACCESS_TOKEN" \ -H "Content-Type: application/json" \ -d '{"value":true}'

3. Criar API Key

curl -X POST http://localhost:3000/api/settings/api-keys \ -H "Authorization: Bearer SEU_ACCESS_TOKEN" \ -H "Content-Type: application/json" \ -d '{"name":"Production Server","permissions":["read","write"]}'

IMPORTANTE: A chave completa é retornada apenas nesta resposta. Armazene imediatamente em local seguro (ex: variáveis de ambiente do servidor).

4. Revogar API Key

curl -X DELETE http://localhost:3000/api/settings/api-keys/abc123-def456 \ -H "Authorization: Bearer SEU_ACCESS_TOKEN"

5. Gerenciar Webhooks

Listar: GET /api/settings/webhooks
Criar: POST /api/settings/webhooks
{"name":"Deploy Hook","url":"https://meu.server/hook","events":["deploy.success","deploy.fail"]}
Testar: POST /api/settings/webhooks/:id/test
Deletar: DELETE /api/settings/webhooks/:id

6. Atualizar Perfil

curl -X PATCH http://localhost:3000/api/profile \ -H "Authorization: Bearer SEU_ACCESS_TOKEN" \ -H "Content-Type: application/json" \ -d '{"name":"João Developer","stack":["React","TypeScript","Node.js"],"cpuLevel":2}'

7. Alterar Senha

curl -X POST http://localhost:3000/api/profile/password \ -H "Authorization: Bearer SEU_ACCESS_TOKEN" \ -H "Content-Type: application/json" \ -d '{"oldPassword":"senhaAntiga123","newPassword":"senhaNovaSegura456"}'

8. Alterar Tema

curl -X PATCH http://localhost:3000/api/profile/theme \ -H "Authorization: Bearer SEU_ACCESS_TOKEN" \ -H "Content-Type: application/json" \ -d '{"theme":"light"}'

9. Deletar Conta

curl -X DELETE http://localhost:3000/api/profile \ -H "Authorization: Bearer SEU_ACCESS_TOKEN"

CUIDADO: Esta ação é irreversível. Todos os dados do usuário serão excluídos (soft delete). O workspace permanece se houver outros membros.

Integrações

🔐

Middleware Auth

Definido em /backend/src/middleware/auth.ts. Todas as rotas de settings e profile usam settingsRouter.use('/*', authMiddleware). O middleware popula c.set('user', { id, email, workspaceId }).

🗄️

Drizzle ORM + DB

Conexão via db singleton. Schemas em /backend/src/db/schema/: users.ts, workspaces.ts, api_keys.ts, webhooks.ts. Queries com eq, and, insert, update, del.

🔑

Crypto (Gerar API Keys)

Função crypto.randomBytes(32).toString('hex') em settings.service.ts. A chave é retornada inteira apenas na criação. O hash é salvo via hashToken() (SHA-256).

📦

Zod Validation

Schemas em /backend/src/schemas/settings.schema.ts e profile.schema.ts. Validação via zValidator('json', schema) do @hono/zod-validator.
