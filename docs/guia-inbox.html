INBOX & NOTIFICATIONS

Central de Mensagens e Notificações

Módulo Inbox & Notifications

Sistema de mensagens centralizado com duas tabelas complementares: inbox_messages (caixa de mensagens do workspace) e notifications (notificações pessoais por usuário).

O que é

Este módulo gerencia dois tipos de mensagens:

  • Inbox Messages: Mensagens do workspace vinculadas a agentes, automações e eventos de sistema.Scoped por workspace.
  • Notifications: Notificações pessoais do usuário, enviadas em tempo real via WebSocket. Scoped por usuário.

Ambos os sistemas permitem marcação como lido, arquivamento e contagem de não lidos.

Arquitetura de Dados

Inbox Messages (Tabela Principal)

Campo Tipo Descrição
id uuid PK - Identificador único
workspaceId uuid FK → workspaces - Escopo do workspace
fromAgentId uuid FK → agents (nullable) - Agente que originou a mensagem
toUserId uuid FK → users - Usuário destinatário
subject varchar(200) Título da mensagem
body text Corpo da mensagem
type enum info | warning | error | success | action
read boolean Status de leitura (default: false)
archived boolean Status de arquivamento (default: false)
actionUrl varchar(500) Link de ação opcional
metadata jsonb Dados adicionais flexíveis
createdAt timestamp Data de criação

Notifications (Tabela de Notificações)

Campo Tipo Descrição
id uuid PK - Identificador único
workspaceId uuid FK → workspaces
userId uuid FK → users - Usuário destinatário
title varchar(200) Título da notificação
body text Corpo da notificação
type enum deploy | agent | system | alert | billing
channel enum in_app | email | push (default: in_app)
read boolean Status de leitura (default: false)
link varchar(500) Link de navegação
createdAt timestamp Data de criação

Notification Preferences

Campo Tipo Descrição
id uuid PK
userId uuid FK → users
type varchar(50) Tipo de notificação (deploy|agent|system|alert|billing)
inApp boolean Notificação in-app (default: true)
email boolean Notificação por email (default: false)
push boolean Notificação push (default: false)

Segurança

  • Workspace Isolation: inbox filtrado por workspaceId do usuário autenticado
  • User Isolation: notifications filtradas por userId do token JWT
  • ActionUrl Validation: Máximo 500 caracteres, implementar validação contra open redirect
  • XSS Prevention: Sanitizar body antes de renderizar no frontend

Riscos Identificados

MÉDIO

Spam de Notificações

Múltiplos agentes/automações podem gerar excesso de mensagens.

Mitigação: Rate limiting por workspace, preferências de notificação
MÉDIO

ActionUrl Open Redirect

Links externos podem redirecionar usuários para sites maliciosos.

Mitigação: Validar domínio, permitir apenas URLs internas ou whitelist
BAIXO

XSS via Body

Corpo da mensagem pode conter HTML/JS malicioso.

Mitigação: Sanitize-html no frontend ao renderizar

Monitoramento

  • GET /api/inbox/unread-count - Contador de mensagens não lidas
  • GET /api/notifications/unread-count - Contador de notificações não lidas

Como Usar

  1. Listar mensagens: GET /api/inbox com filtros opcionais (type, read, archived)
  2. Ver detalhe: GET /api/inbox/:id
  3. Marcar como lido: POST /api/inbox/:id/read
  4. Marcar todos como lidos: POST /api/inbox/read-all
  5. Arquivar: POST /api/inbox/:id/archive (marca como lido automaticamente)
  6. Desarquivar: POST /api/inbox/:id/unarchive
  7. Excluir: DELETE /api/inbox/:id
  8. Gerenciar preferências: GET/PUT /api/notifications/preferences
