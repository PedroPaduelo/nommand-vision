Status & Monitoring

Painel de saúde dos serviços com monitoramento de uptime, histórico de incidentes, métricas de disponibilidade e alertas proativos.

📋 Visão Geral

Propósito

Monitorar saúde dos serviços em tempo real, rastrear uptime por serviço, gerenciar incidentes e prover visibilidade sobre disponibilidade do sistema.

Arquitetura

Hono + Drizzle ORM + PostgreSQL. Health checks registrados via recordCheck() com histórico em tabela dedicada.

Estado

Stable Implementação completa com endpoints CRUD para serviços e incidentes.

Tech Stack

TypeScript, Zod validation, Drizzle ORM, React hooks, CSS modules

🔗 Rotas API

Todas as rotas requerem autenticação via authMiddleware e filtram por workspaceId.

GET
/api/status

Overview de todos os serviços do workspace. Retorna {globalStatus, services[]} com status agregado.

GET
/api/status/services/:id

Obtém serviço específico com todos os detalhes.

POST
/api/status/services

Cria novo serviço para monitoramento. Schema: createServiceSchema.

PATCH
/api/status/services/:id

Atualiza configurações do serviço. Schema: updateServiceSchema (campos opcionais).

DELETE
/api/status/services/:id

Remove serviço (cascade: incidentes e checks). Retorna {success: true}.

GET
/api/status/services/:id/uptime?days=30

Histórico de uptime do serviço. Query: days (1-90, default 30). Retorna {history[], percentage, total, up}.

GET
/api/status/incidents

Lista incidentes do workspace. Query opcional: status, severity, serviceId. Schema: listIncidentsSchema.

POST
/api/status/incidents

Cria novo incidente. Schema: createIncidentSchema.

PATCH
/api/status/incidents/:id

Atualiza incidente (título, descrição, severidade, status). Schema: updateIncidentSchema.

POST
/api/status/incidents/:id/resolve

Marca incidente como resolved e define resolvedAt automaticamente.

⚙️ Status Service

Arquivo: backend/src/services/status.service.ts

Funções de Serviço

getOverview(workspaceId)

Retorna status global e lista de serviços. Calcula globalStatus: 'operacional' se todos up, 'degraded' se algum degraded, 'major_outage' se algum major_outage.

getService(id)

Busca serviço por ID. Throws 'Service not found' se não existir.

createService(workspaceId, input)

Cria novo serviço. Valores default: type='api', status='operational'.

updateService(id, input)

Atualiza campos do serviço. Campos são parciais. Atualiza updatedAt automaticamente.

deleteService(id)

Remove serviço (CASCADE para incidentes e checks). Retorna {success: true}.

recordCheck(serviceId, status, responseTimeMs)

Não documentado nas rotas públicas - função interna para health checks. Insere registro em uptimeChecks e atualiza services.status e lastCheckAt. Mapeia: 'up'→'operational', 'degraded'→'degraded', 'down'→'major_outage'.

getUptimeHistory(serviceId, days = 30)

Retorna array de checks dos últimos N dias ordenado por checkedAt DESC.

getUptimePercentage(serviceId, days = 30)

Calcula porcentagem deuptime (checks com status='up'). Retorna {percentage, total, up}.

listIncidents(workspaceId, filters)

Lista incidentes com filtros opcionais: status, severity, serviceId. Ordenado por startedAt DESC.

createIncident(workspaceId, input)

Cria incidente. Default: severity='minor', status='investigating'. startedAt é definido automaticamente.

updateIncident(id, input)

Atualiza campos do incidente. Atualiza updatedAt.

resolveIncident(id)

Marca incidente como status='resolved' e define resolvedAt como data atual.

🔒 Segurança

Controle de Acesso

  • Workspace isolation: Todas as queries filtram por workspaceId via authMiddleware.
  • Auth required: Todas as rotas usam statusRouter.use('/*', authMiddleware).
  • FK Cascade Delete: Deleção de serviço remove automaticamente incidentes e checks relacionados.

Riscos Identificados

🟡 Health Check Endpoint Exposto

Função recordCheck existe no service mas não há rota pública para receber health checks. Isso significa que não é possível integrar com sistema de monitoramento externo (Statuspal, UptimeRobot, etc).

Mitigação: Expor endpoint POST /api/status/checks/record com API key ou token de serviço.
🟡 Sem Rate Limiting Específico

Health checks podem ser spamados. Rotas de status usam authMiddleware mas não tem rate limiting dedicado por endpoint.

Mitigação: Implementar rate limit mais leniente para health checks (ex: 60 req/min por workspace).
🟢 histórico Ilimitado

Tabela uptimeChecks cresce indefinidamente. Sem política de retenção ou particionamento por tempo.

Mitigação: Implementar TTL de 2 anos ou particionamento mensal, agregar checks antigos.

📖 Como Usar

Para Operadores / Admin

  1. Visualizar painel em /status (frontend página Status).
  2. Configurar serviços custom: adicionar endpoints externos (ex: API pública, DB externo, worker).
  3. Criar incidentes manualmente quando problemas forem detectados (ex: deploy falhou).
  4. Atualizar status dos serviços conforme investigação progride.
  5. Resolver incidentes quando problema estiver sanado.
  6. Usar gráfico de uptime para identificar padrões de indisponibilidade.

Para Desenvolvedores Backend

  1. Integrar Health Checks: Chamar await recordCheck(serviceId, 'up'|'degraded'|'down', responseTime) após cada verificação.
  2. Criar endpoint de health check: Em seu serviço, expor GET /health que retorna 200/500 e measure response time.
  3. Agendar monitoramento: Configurar cron job externo ou scheduler interno para poll dos endpoints registrados.
  4. Alertas automatizados: Integrar com automations para POST /api/status/incidents quando status != 'operational'.

Para Desenvolvedores Frontend

  1. Consumir GET /api/status para painel principal.
  2. Consumir GET /api/status/services/:id/uptime?days=30 para gráfico de barras.
  3. Consumir GET /api/status/incidents com busca por texto para histórico.
  4. Botões CRUD chamam respectivos endpoints (POST/PATCH/DELETE) com autenticação.
  5. Ver hooks em sistema-react/src/hooks/queries/useStatusQueries.ts.
